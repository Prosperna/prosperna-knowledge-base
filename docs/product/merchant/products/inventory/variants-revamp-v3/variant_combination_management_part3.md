#### **UC 42 | Auto-Calculate Variant Combination Margin**

| **Field**           | **Description**                                                                                                                                                         |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Use Case ID         | UC 42                                                                                                                                                                   |
| Prepared By         | BA                                                                                                                                                                      |
| Last Updated        | September 18, 2025                                                                                                                                                      |
| Objectives          | The system should automatically calculate and display the profit margin for a variant combination based on a specific hierarchy of price fields.                        |
| Actor               | System                                                                                                                                                                  |
| Preconditions       | The merchant enters, changes, or removes a value in the Price, Sale Price, or Unit Cost field for a variant combination.                                                |
| Steps               | 1\. The system continuously monitors the Price, Sale Price, and Unit Cost fields for any change.                                                                        |
|                     | 2\. Upon a change, the system calculates the Margin value based on the following hierarchy:                                                                             |
|                     | **Condition 1:** **`Sale Price`** **and** **`Unit Cost`** **have values.**<br />a. The system calculates `Margin = Sale Price - Unit Cost`.                             |
|                     | **Condition 2:** **`Sale Price`** **has a value (and** **`Unit Cost`** **is empty).**<br />a. The system sets `Margin = Sale Price`.                                    |
|                     | **Condition 3:** **`Price`** **and** **`Unit Cost`** **have values (and** **`Sale Price`** **is empty).**<br />a. The system calculates `Margin = Price - Unit Cost`.   |
|                     | **Condition 4:** **`Price`** **has a value (and** **`Sale Price`** **and** **`Unit Cost`** **are empty).**<br />a. The system sets `Margin = Price`.                    |
|                     | **Condition 5:** **`Price`\*\***,\*\* **`Sale Price`\*\***, and\*\* **`Unit Cost`** **are all empty.**<br />a. The system sets `Margin = 0`.                            |
|                     | 3\. The system displays the newly calculated value in the non-editable "Margin" column for that row, formatted to two decimal places.                                   |
| Postconditions      | The "Margin" column always shows the correct, real-time profit margin based on the defined calculation hierarchy.                                                       |
| Business Trigger    | To provide merchants with immediate and accurate insight into the profitability of their pricing decisions under different scenarios (e.g., on sale vs. regular price). |
| Acceptance Criteria | If Sale Price and Unit Cost exist, Margin is `Sale Price - Unit Cost`.                                                                                                  |
|                     | If Sale Price exists (but no Unit Cost), Margin is `Sale Price`.                                                                                                        |
|                     | If Price and Unit Cost exist (but no Sale Price), Margin is `Price - Unit Cost`.                                                                                        |
|                     | If only Price exists, Margin is equal to Price.                                                                                                                         |
|                     | If Price, Sale Price, and Unit Cost are all empty, Margin is 0.                                                                                                         |
|                     | The Margin value updates instantly whenever a constituent field (Price, Sale Price, Unit Cost) is changed.                                                              |
| Estimates           |                                                                                                                                                                         |
| Success Message     |                                                                                                                                                                         |
| Error Messages      |                                                                                                                                                                         |

| **Business Rules/Desired Behavior**                                                                                                                                                                                                           |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| - The Margin field itself is not editable. It is always a calculated value.<br />- The Margin value can be a negative number if the Unit Cost is higher than the Sale Price or Price value; this scenario should not be blocked by the system |

#### **UC 50 | Handle Plan Downgrade with Soft Enforcement**

| **Field**           | **Description**                                                                                                                                                                                           |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Use Case ID         | UC 50                                                                                                                                                                                                     |
| Prepared By         | BA                                                                                                                                                                                                        |
| Last Updated        | October 26, 2025                                                                                                                                                                                          |
| Objectives          | The system should implement soft enforcement when a merchant downgrades their plan, preserving existing variant data that exceeds new limits while preventing new additions beyond the downgraded limits. |
| Actor               | System, Merchant                                                                                                                                                                                          |
| Preconditions       | The merchant has downgraded their subscription plan.                                                                                                                                                      |
|                     | The merchant has existing variant data (types, options, or combinations) that exceeds the limits of their new plan.                                                                                       |
| Steps               | **Scenario 1: Downgrade with Variant Types Exceeding New Limit**                                                                                                                                          |
|                     | 1\. A merchant downgrades from Premium (6 types) to Pro (4 types) with 5 existing variant types.                                                                                                          |
|                     | 2\. The system preserves all 5 existing variant type cards and their associated data.                                                                                                                     |
|                     | 3\. The system hides the "Add Variant Type" button.                                                                                                                                                       |
|                     | 4\. If the merchant attempts to add a new variant type (via any method), the system displays an error toast: "4 Variant type limit reached."                                                              |
|                     | 5\. The merchant can edit or delete existing variant types within the current count.                                                                                                                      |
|                     | 6\. Once the merchant manually reduces types to 4 or below, the "Add Variant Type" button reappears.                                                                                                      |
|                     | **Scenario 2: Downgrade with Options Exceeding New Limit**                                                                                                                                                |
|                     | 1\. A merchant downgrades from Pro (40 options) to Plus (30 options) with a variant type containing 35 options.                                                                                           |
|                     | 2\. The system preserves all 35 existing options and their associated combinations.                                                                                                                       |
|                     | 3\. The system hides the placeholder option field for that variant type.                                                                                                                                  |
|                     | 4\. If the merchant attempts to add a new option, the system prevents the action (no placeholder visible).                                                                                                |
|                     | 5\. The merchant can edit or delete existing options.                                                                                                                                                     |
|                     | 6\. Once the merchant manually reduces options to 30 or below, the placeholder option field reappears.                                                                                                    |
|                     | **Scenario 3: Downgrade with Combinations Exceeding New Limit**                                                                                                                                           |
|                     | 1\. A merchant downgrades from Premium (150 combinations) to Free (50 combinations) with 120 existing combinations.                                                                                       |
|                     | 2\. The system preserves all 120 existing combinations and their associated data (prices, SKUs, images, etc.).                                                                                            |
|                     | 3\. All 120 combinations remain visible, editable, and sellable on the storefront.                                                                                                                        |
|                     | 4\. The system hides placeholder option fields for all variant types (since adding any option would exceed 50).                                                                                           |
|                     | 5\. The system hides the "Add Variant Type" button.                                                                                                                                                       |
|                     | 6\. The merchant can edit existing variant data but cannot add new types, options, or combinations.                                                                                                       |
|                     | 7\. If the merchant deletes variant types or options to reduce total combinations to 50 or below, the system re-enables adding based on the 50-combination constraint.                                    |
| Postconditions      | All existing variant data is preserved and remains functional.                                                                                                                                            |
|                     | The merchant cannot add new variant types, options, or combinations beyond the new plan's limits.                                                                                                         |
|                     | The merchant can edit or delete existing data within the preserved set.                                                                                                                                   |
|                     | Products with preserved variants remain sellable on the storefront without disruption.                                                                                                                    |
| Business Trigger    | The system must gracefully handle plan downgrades without causing immediate disruption to live products while still enforcing subscription limits for future modifications.                               |
