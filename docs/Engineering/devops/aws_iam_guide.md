# Create Secure IAM Guide

This guide provides step-by-step instructions and best practices for managing AWS IAM (Identity and Access Management) securely in Prosperna. It covers how to structure IAM users, groups, policies, access control, auditing, and key rotation.

## 📌 Overview

At Prosperna, we use AWS IAM to manage user permissions across our infrastructure. Each employee is granted access based on their job role, using IAM groups with role-based policies. We follow the principle of least privilege and regularly audit access.

---

## 👥 IAM Users and Groups

### Steps to Add a New Employee

1. Go to [AWS IAM Console](https://console.aws.amazon.com/iam/).
2. Click **Users** > **Add users**.
3. Enter username: `juan.delacruz@prosperna.com`
4. Select **Access type**:
   - AWS Management Console access: ✅
   - Autogenerated password, require password reset on first sign-in
5. Click **Next: Permissions**
6. Select **Add user to group**
7. Choose from one of the predefined groups:
   - `DevOps`
   - `Frontend-Developers`
   - `Backend-Developers`
   - `devops-intern`
   - `QA`
8. Click **Next** until **Create user**

### IAM Group Example: Developers

- **Group Name:** `Developers`
- **Attached Policies:**
  - `AmazonEC2ReadOnlyAccess`
  - `AmazonS3ReadOnlyAccess`
  - Custom policy for project-specific S3 and DynamoDB access

---

## 🔐 IAM Policy Structure

All groups should use **managed policies** with restricted, specific permissions.

### Sample Policy (Least Privilege)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::prosperna-dev-resources",
        "arn:aws:s3:::prosperna-dev-resources/*"
      ]
    }
  ]
}
```

## ✅ IAM Best Practices

| Best Practice                  | How to Implement                                                     |
| ------------------------------ | -------------------------------------------------------------------- |
| **Use Groups for Permissions** | Do not assign policies directly to users. Create IAM groups by role. |
| **Enable MFA**                 | Require MFA for all console users.                                   |
| **Enforce Least Privilege**    | Regularly review policies. Only grant necessary actions.             |
| **Password Policy**            | Use strong password requirements: 12+ characters, symbols, numbers.  |
| **Rotate Keys Regularly**      | Set reminders to rotate access keys every 90 days.                   |
| **Monitor IAM Access**         | Use AWS CloudTrail and AWS Config to audit changes.                  |

## 🔁 Key Rotation & Auditing

### Key Rotation

1. Go to IAM Console > **Users**
2. Click on a user: `juan.delacruz@prosperna.com`
3. Go to **Security Credentials**
4. Under **Access Keys**, click **Create access key**
5. Replace old key in environment/configs
6. Delete old access key

### Audit Access

- Enable **CloudTrail** in all regions
- Use **AWS Config** rules like:
  - `iam-user-no-policies-check`
  - `iam-group-has-users-check`
- Periodically run IAM **Credential Report**
- Set up CloudWatch alarms for suspicious activity

## 🧪 Example Use Case: New DevOps Engineer

> Scenario: A new DevOps Engineer joins Prosperna

1. Add user: `paski.devops@prosperna.com`
2. Assign to group: `DevOps`
3. Group policies include:
   - `AmazonEC2FullAccess`
   - `CloudWatchFullAccess`
   - Custom SSM and Secrets Manager access
4. Send onboarding email with temporary password
5. Confirm MFA is enabled

## ✅ Summary

IAM is the backbone of security in AWS. At Prosperna, we:

- Use role-based IAM groups
- Enforce least privilege
- Rotate keys
- Audit regularly

> 🔐 Always treat IAM like production infrastructure — version-controlled, monitored, and locked down by default.
